# 深度思考路由功能设计

## Situation — 情景

在 RAGStudio 项目中，用户可以启用深度思考（Deep Reasoning）让大模型处理更复杂的任务。然而，**不同 LLM 厂商对推理深度的 API 设计截然不同**：

- **离散型（Discrete）** ：如 DeepSeek-R1，推理强度是固定的几档（`off / low / medium / high`），没有中间值
- **连续型（Continuous）** ：如 Claude Opus 4、Gemini 2.5 Pro，通过 Token 预算（`budget_tokens`​ / `thinkingBudget`）控制思考深度，值域是一个连续的整数范围
- **布尔型（Boolean）** ：如 Ollama 部署的 Qwen3，只支持**开**或**关**，没有渐变
- **无支持（None）** ：如 GPT-4o，根本不暴露推理深度参数

这种异构性导致了一个根本性的 UX 问题：前端必须为每种模型单独设计控制 UI，用户也必须在切换模型时了解其底层的 API 语义——**没有一个统一的控制方案**。

‍

## Task — 任务

设计并实现一个**统一的推理深度路由层**，满足三个目标：

1. **前端**：无论后端激活哪个模型，用户始终面对同一个 0–100 的滑块，操作直觉一致
2. **后端**：将滑块值透明地映射为各模型原生的 API 参数，用户无需关心底层差异
3. **架构**：高度可扩展——新增模型或厂商只需修改配置，核心适配逻辑零改动

核心设计哲学为**归一化与反归一化**（Normalization & Denormalization）：

> 前端滑块 0–100  →  内部归一化 0.0–1.0  →  反归一化为各模型的 API 参数

其中 **0 是全局通用的"关闭"信号**，不参与任何映射逻辑。

‍

## Action — 行动

### 1. Model Registry（模型注册表）

声明式注册表，将每个模型的推理能力归为四种类型之一。每条记录定义了归一化区间 `(0, 1]` 如何映射到该模型的 API。

```json
[
  {
    "modelId": "claude-opus-4",
    "provider": "anthropic",
    "reasoningSupport": {
      "type": "CONTINUOUS",
      "config": {
        "minTokens": 1024,
        "maxTokens": 8192,
        "step": 1
      }
    }
  },
  {
    "modelId": "gemini-2.5-pro",
    "provider": "google",
    "reasoningSupport": {
      "type": "CONTINUOUS",
      "config": { "minTokens": 128, "maxTokens": 4096, "step": 128 }
    }
  },
  {
    "modelId": "deepseek-r1",
    "provider": "deepseek",
    "reasoningSupport": {
      "type": "DISCRETE",
      "config": {
        "thresholds": [
          { "value": 0.33, "param": { "effort": "low" } },
          { "value": 0.66, "param": { "effort": "medium" } },
          { "value": 1.0,  "param": { "effort": "high" } }
        ]
      }
    }
  },
  {
    "modelId": "qwen3-235b",
    "provider": "ollama",
    "reasoningSupport": {
      "type": "BOOLEAN",
      "config": { "enableParam": "enable_thinking" }
    }
  },
  {
    "modelId": "gpt-4o",
    "provider": "openai",
    "reasoningSupport": { "type": "NONE" }
  }
]
```

### 2. ReasoningAdapter（核心适配器）

适配器的入口 `adapt(rawValue, modelConfig)` 遵循清晰的三步流水线：

> rawValue (0–100) → ① 零值守卫 → ② 归一化(0.0–1.0) → ③ 按类型分发 → API 参数

**① 零值守卫**：若 `rawValue == 0`，直接返回该厂商的"关闭" Payload，不进入后续逻辑。这是全局统一的关闭开关，代码路径最短、最安全。

**② 归一化**：`progress = rawValue / 100.0`​。经过零值守卫后，progress 必定 `> 0`。

**③ 按类型分发**：

|类型|策略|示例|
| ------| ---------------------------------------------------------| ----------------------------------|
|`CONTINUOUS`|线性插值：`minTokens + (maxTokens - minTokens) × progress`，对齐步长、夹紧边界|Claude 滑块 50 → `budget_tokens = 1024 + (8192 - 1024) × 0.5 = 4608`|
|`DISCRETE`|阈值查找：遍历 thresholds，找到首个 `progress ≤ threshold.value` 的项作为当前挡位|DeepSeek 滑块 50 → `progress = 0.5`​，`0.5 ≤ 0.66`​ → `effort: "medium"`|
|`BOOLEAN`|能走到这个分支说明 progress > 0，直接返回开启参数|Qwen3 → `enable_thinking: true`|
|`NONE`|返回空对象|GPT-4o → `{}`|

**连续型插值公式**：

```
budget = minTokens + (maxTokens - minTokens) × progress
budget = ceil(budget / step) × step          // 向步长对齐
budget = clamp(budget, minTokens, maxTokens) // 边界保护
```

**离散型阈值查找**：

```
selected = thresholds[last]   // 默认取最高挡
for each threshold:
    if progress ≤ threshold.value:
        selected = threshold.param
        break
```

**厂商关闭逻辑的集中管理**：

```
if rawValue == 0:
    anthropic → {"thinking": {"type": "disabled"}}
    google   → {"generationConfig": {"thinkingConfig": {"thinkingBudget": 0}}}
    deepseek → {"enable_thinking": false}
    ollama   → {"chat_template_kwargs": {"enable_thinking": false}}
```

### 3. 前端交互增强

- **0 位置标签**："关闭思考"，明确告知用户当前不启用推理
- **1–99 位置标签**："深度思考 (x%)"，直观展示当前深度
- **动态 Tooltip**：连续型模型显示"预估思考 Token: ~4600"；离散型模型显示"挡位：中等思考"
- **可选的磁吸效果**：离散型模型在阈值边界（33、66）产生轻微吸附感，帮助用户在挡位切换处精准定位

## Result — 结果

该架构在 RAGStudio 中实现后，取得了以下效果：

### 1. 完美的归一化

`0–100 → 0.0–1.0`​ 成为系统内部唯一的控制语言。前端代码**零分支判断模型类型**，无论后端切换到 Claude、DeepSeek 还是 Qwen3，滑块的行为和语义完全一致。新增模型时，前端无需任何改动。

### 2. 优雅的零值处理

全局 `rawValue == 0`​ 守卫使"关闭"逻辑集中在单个入口，避免了连续型模型意外收到 `budget_tokens: 0` 导致 API 报错的风险。代码审查时，零值路径一目了然。

### 3. 离散映射灵活且符合直觉

`thresholds` 数组支持任意分段策略——既可以是三等分（33/66/100），也可以是二八法则（20/100），甚至定义 10 级精细挡位。所有分段逻辑是声明式的配置，核心适配代码零修改。

### 4. 连续映射安全可靠

- `minTokens` 保护了底层 API 的硬性下限（如 Claude 要求 ≥1024），杜绝 API 400 错误
- `maxTokens` 防止无限制的 Token 预算膨胀
- `step` 对齐确保输出值始终是 API 接受的合法值（如 Gemini 要求 128 的倍数）

### 5. 极高的可扩展性

|场景|改动范围|
| ---------------------------| ----------------------------------------|
|新增一个 10 挡离散模型|`thresholds` 加 9 行配置|
|新增一个连续型模型|Registry 加一条记录，填写 `minTokens / maxTokens / step`|
|新增一个厂商（如 Cohere）|`_disable_reasoning` 开关加一个分支 + Payload 构建函数|
|调整某个模型的挡位分界|修改该模型的 `thresholds.value` 即可|

核心适配逻辑（归一化、插值、阈值查找）零改动。

### 6. 清晰的关注点分离

- **Registry** 是纯数据——描述"是什么"
- **Adapter** 是纯转换逻辑——执行"怎么映射"
- **Frontend** 是纯 UX——负责"怎么呈现"

三层可独立演化、独立测试，任一层的变更不会波及其他层。这套架构已成为 RAGStudio 中所有推理模型接入的标准范式。
